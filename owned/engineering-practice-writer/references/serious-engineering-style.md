# Serious Engineering Style

在用户要求“更像架构解释文档”“偏底层机制说明”“像 V8 / React Native 这类工程文档”时，读取本文件。

本文档提炼的是一组偏架构解释、机制深挖、系统说明的官方工程文档共同写法，而不是要求复制标题、章节名或平台术语：

- React Native `Architecture Overview`
- React Native `Fabric`
- React Native `Render, Commit, and Mount`
- React Native `Threading Model`
- V8 `Sparkplug`
- V8 `Maglev`
- V8 `Pointer Compression in V8`

## 总体判断

这类文章的核心目标不是复盘一次实践，也不是提交一个待评审提案，而是把一个系统、一个运行机制或一段架构分层讲清楚。它既不像 RFC 那样强调 proposal 结构，也不像技术博客那样强调对象如何在实践中一步步演进。

更准确地说，这类文风是在回答三类问题：

- 这个对象在系统里处在什么位置
- 为什么需要这样一层设计或这样一组约束
- 它具体是如何工作的

## 可执行的风格维度

### 1. 先立系统位置，再谈细节

- 开头通常先告诉读者：对象是什么，它位于哪一层，它解决的是哪一类问题。
- 背景会有，但背景主要服务于理解架构位置，而不是讲一段完整的实践故事。

迁移规则：

- 如果草稿一开始就在讲实现细节，先补“这个对象在整体里扮演什么角色”。
- 如果草稿一上来就在讲个人经历或过程，先收回到系统位置和设计动机。

### 2. 先有概念框架，再做分层展开

- 这类文章通常会先给读者一个总的架构框架，再逐层往下拆。
- 常见写法是先给出系统图景、层次关系或阶段划分，然后再解释每层的职责和交互。

迁移规则：

- 长文优先先总后分，不要让读者在进入细节前还不知道整体框架。
- 如果对象天然分层，显式写出层次和边界。

### 3. 主体是机制解释，不是方案回顾

- 正文重心通常落在数据流、执行阶段、线程模型、编译层级、对象关系、状态转换等机制上。
- 文章推进主要依赖“对象定义 -> 约束说明 -> 机制展开 -> 边界补充”。

迁移规则：

- 少写“我们后来怎么做”，多写“这个机制为什么需要这样设计，以及它如何工作”。
- 如果出现历史信息，让它只服务于理解当前设计，不要让正文退化成演进史。

### 4. 术语稳定，定义先行

- 关键对象和术语通常会被稳定使用，不会频繁换说法。
- 当一个概念容易混淆时，文章会先定义它，再继续讲下去。

迁移规则：

- 不要同一个对象一会儿叫“链路”，一会儿叫“流程”，一会儿叫“框架”，除非这些词真有不同语义。
- 第一次出现核心对象时，优先补一句“它是什么/不是什么”。

### 5. 结构常带明显的阶段感或层次感

- React Native 的 render pipeline、threading model，V8 的 tiering、IR、representation，都有很强的阶段或层次结构。
- 这类文章常把一段机制拆成若干阶段或若干层，让读者沿着结构读，而不是沿着叙事读。

迁移规则：

- 如果对象有明确阶段，优先按阶段写。
- 如果对象有明确层次，优先按层次写。
- 不要把阶段、层次、边界全混在一个段落里。

### 6. 取舍和限制要出现，但不写成 RFC

- 这类文章会谈约束、限制、实现代价和边界条件，但不会像 RFC 一样强制拉出 `Drawbacks`、`Alternatives`、`Adoption`。
- 这些信息通常自然嵌入在机制解释里。

迁移规则：

- 要写边界和约束，但不必硬拆成 proposal 章节。
- 更自然的写法是：“这样设计的原因是……，代价是……，边界在于……”。

### 7. 文风偏“架构解释语体”

- 词汇通常偏：架构、层、阶段、线程、表示、约束、语义、执行、调度、编译、对象、状态、接口、同步、异步、共享、不可变、优化、开销。
- 常见连接方式是：从整体上看、本质上、这里的关键在于、进一步看、在这一层、对应地、因此、与此同时。
- 句子可以偏中长，但应保持清晰的层级和从句关系，让读者顺着机制往下走。

迁移规则：

- 优先使用解释型连接词，而不是博客叙事型连接词。
- 少用“我们”“后来”“当时”，除非它们确实是解释设计决策所必需的。
- 把抽象判断改成对象、层次和约束绑定的判断。

## 与其他风格的边界

- 它不是 `practice-sharing`：不以一条实践演进线为主，不依赖个人/团队视角推动阅读。
- 它不是 `rfc-document`：不以评审、proposal、rollout、alternatives 为默认骨架。
- 它更像一篇“带读者理解系统内部结构”的工程解释文。

## 不要做的事

- 不要把 serious-engineering 写成实践复盘。
- 不要把 serious-engineering 写成 RFC 模板。
- 不要跳过对象定义，直接进入实现细节。
- 不要为了显得专业而堆砌抽象术语，导致对象和层次反而更模糊。
- 不要让术语在文中频繁漂移。

## 快速判断

如果改写后的文本满足下面这些特征，通常就更接近 serious-engineering：

- 读者很快知道对象在系统中的位置
- 正文能先给框架，再给分层或分阶段展开
- 机制和约束是文章主体，而不是实践经历
- 术语稳定，关键概念先定义再展开
- 风格克制、解释性强、结构感明确
- 不像博客，也不像 RFC，而像一篇成熟的架构/机制说明文
