# Research Report Style

在用户要求“写调研报告”“做技术选型调研”“围绕一个 URL 或主题生成中文研究报告”“做技术方案对比”时，读取本文件。

这类文本的目标不是做叙事表达，也不是提交评审提案，而是帮助读者完成一件更具体的事：

- 建立对研究对象的清晰认识
- 理解当前资料能支持到什么程度
- 形成可执行的 first-pass 验证路径
- 判断适用边界、主要风险和下一步动作

换句话说，它更接近“面向判断与落地的研究整理”，而不是“面向表达的技术文章”。

## 可借鉴的范式

这类写法更适合借下面几类范式，而不是技术博客：

- AWS Decision Guides
  - 长处在于先交代 `Purpose`，再把对象、判断标准、候选路径和 next step 组织成一条自然的决策线。
- Azure Technology Choice / Decision Guides
  - 长处在于用 flowchart、比较表和 team-skill / operational-overhead 维度，把“适合 / 不适合 / 成本 / 限制”说清楚。
- Thoughtworks Technology Radar
  - 长处在于 recommendation tier 非常清楚，结论短、判断硬，但不会把全部实现细节塞进结论摘要。

迁移规则：

- 借这些材料的判断组织方式，不要照抄它们的章节名。
- 对单项目调研，通常不需要把正文写成完整 decision tree，但要借它们“先给判断、再给依据、再给 next step”的顺序。
- 对单项目评估，一个更自然的正文骨架通常是：
  - 范围与结论
  - 它是什么
  - 它解决什么问题
  - 它是怎么组织起来的
  - 怎么开始试一轮
  - 采用边界与主要风险
  - 下一步建议
- 对工具型、平台型、服务型项目，还可以借鉴一种更轻量的真人调研 memo 气质：
  - 先用产品角色把对象说清楚
  - 如果已经试过，尽早给出一小段 `实测`
  - 把 `观察` 和 `评估` 分开
  - 允许用一句克制的类比帮助读者建立直觉，但不要让类比替代判断
  - 正文先从产品层和使用层展开，再用实现层内容支撑关键判断

## 默认要回答的问题

一篇像样的调研报告，通常都应该回答下面这些问题：

- 调研对象到底是什么，本次范围到哪里为止
- 当前结论主要建立在什么资料边界上
- 如果只做 first pass，最小 SOP 是什么
- 它更适合哪些场景，不适合哪些场景
- 主要坑、限制、配置成本和前置条件在哪里
- 基于当前证据，最合理的建议或下一步是什么

如果正文没有实质回答这些问题，再长也更像资料摘要，而不是调研报告。

## 结论摘要应该怎么写

项目调研里最容易写坏的部分，就是“先看结论”。好的结论摘要，不该只是把若干实现层判断压成一句抽象定义，而应该先回答读者最关心的采用问题。

更稳的写法通常包含下面几层，而且顺序尽量固定：

- 先用读者熟悉的诉求语言解释它在解决什么问题
  - 例如先回答“如果你的目标是统一接入、快速验证、降低部署门槛、减少运维复杂度，这个对象有没有研究价值”
- 先用读者熟悉的类别解释对象
  - 例如先回答“它更像什么工具 / 平台 / 运行层 / 接入层”，再补更抽象的机制定义
- 再给一个有边界的 recommendation tier
  - 如“值得做 PoC”“可以先试”“暂时观望”“当前不建议作为首选”
- 再写 best fit / weak fit
  - 它更适合什么目标，不适合什么目标
- 再补 first-pass 路径
  - 如果要验证，最短链路是什么
- 最后补 evidence boundary
  - 当前结论是 docs-level、docs + code-reading，还是已经过实测

迁移规则：

- 这些层次是写作者内部的检查框架，不是要逐项写到表面。结论摘要应该像已经消化过材料后的自然判断，而不是把模板逐行念出来。
- 结论摘要优先写成 2 到 3 个凝练判断段，而不是一串模板化的“如果你的诉求是……”句子。
- 先给综合后的主判断，再补适用边界和验证边界；不要让结论读起来像使用说明书。
- 结论摘要不要承载过多操作顺序或验证步骤细节。验证策略应主要放到后面的“最小 SOP”“验证路径”或“建议与下一步”部分。
- 不要把结论摘要写成“它本质上是一个……”然后直接进入架构术语。
- 不要一上来堆项目专有名词、接口名、实现名词或内部结构词。先用普通读者能立即理解的诉求语言和判断语言。
- 不要让实现层分类词独自承担整个结论，例如只写成“统一 API 控制面”“编排层”“中间层”，却不先解释这对采用者意味着什么。
- 不要优先使用“控制面”“平台层”“中间层”这类抽象架构词来定义对象。优先用更接近采用判断的对象角色，例如“网关”“接入层”“服务框架”“统一入口”。
- 先说“如果你想解决什么问题，它有没有价值”，再说“它对谁更像什么、适合什么场景”，最后再说“从实现上看它为什么会这样”。
- 结论摘要可以短，但必须让读者立刻知道：该不该继续看、该不该试、试什么。
- 避免连续使用相同句式起手，例如反复用“如果你的目标是……”“这意味着……”“对这次调研来说……”。成熟的调研文更像自然收束的 memo，而不是同一模板句的排列组合。
- 避免把 `适合场景 / 不适合场景 / recommendation / evidence boundary / 仍需验证` 压进一句过长的让步复句里。优秀的调研文通常会把主判断、适用边界和证据边界拆成两到三个短段或短句，而不是一口气说完。
- 避免元话术起手，例如“如果先只回答一个问题”“更稳妥的说法是”“这轮调研不打算下结论”。直接写判断本身，通常会更像成熟报告。
- 证据边界也不要写成抽象自我说明，例如“目前还停留在……这一层”“足以说明……但还不能替代……”。更自然的写法是直接说明：这轮已经确认了什么，还有哪些关键问题需要靠实测补上。
- 避免把证据边界写成研究过程回顾，例如“这轮能确认的是……”“还没有回答的是……”。读者更关心对象本身和当前可判断的边界，而不是作者此刻的研究姿态。

## 组织原则

### 0. 让判断框架藏在文内，不要浮到表面

- 调研报告当然需要有清楚的判断维度，但这些维度应该作为隐式骨架存在，而不是直接变成生硬的标题、段首模板和重复句式。
- 读者应该先感受到“这是一篇成熟的项目判断”，而不是“作者在按表填写结论、SOP、场景、风险”。
- 一旦对象角色和核心问题已经在前文立住，后文就应该继续推进新信息，而不是在每节开头重复定义同一个对象。

迁移规则：

- 不要为了完整覆盖维度，就把每个维度都独立成一个机械章节。
- 相近维度可以合并，例如“适用场景 / 不适用场景 / 主要风险”常常可以在同一节里自然展开。
- 标题优先像真实读者会问的问题，或像对象本身的自然分面，而不是像内部写作模板。
- 小节标题尽量自足，不要依赖上一段结论才能读懂。少用“这个判断为什么成立”“为什么这么看”“这个结论意味着什么”这类回指式标题。

### 1. 先立对象，再立判断

- 开头优先说清楚研究对象、研究问题和资料边界。
- 对具名项目、产品、库或框架，优先先回答三个最基础的问题：它是什么、它是干什么的、它主要解决什么问题。
- 对工具型、平台型、服务型项目，优先从产品角色和使用入口建立直觉。必要时先交代官网、主要入口、部署方式、是否有 WebUI / CLI / API 等使用面，再进入更深的机制解释。
- 如果对象本身可以被一句话说清楚，这句话通常就应该成为 opening 的第一句。不要先用范围说明、资料边界或研究过程起手。
- 开篇优先停留在对象、用途和核心问题上。像硬件门槛、特定能力路径成熟度、性能边界这类细节，除非它本身就是对象定义的一部分，否则放到后文对应章节展开。
- 先回答“研究什么、想回答什么、依据来自哪里”，再进入展开。
- 如果用户点名了一个项目、产品、库或框架，默认范围就是这个对象本身，不要自动扩成整个赛道。
- 表达 scope 时优先用正向说法，例如“本次调研聚焦 X 项目本身”，而不是先写“不扩展到更广泛的 Y 赛道”这类范围防御句。
- 不要在 opening 里一次性堆满 `scope + sources + 没做什么 + 结论强度`。更自然的方式是先给研究对象和核心问题，再用一句较短的话交代资料边界与结论强度。
- 对单项目调研，opening 尽量少写“本文重点回答三个问题”“这份调研围绕 X 展开”这类作者过程说明。优先直接写对象定义和本文真正关心的判断。
- 对具名项目调研，一个更自然的 opening 顺序通常是：一句话对象描述 -> 当前主判断 -> 后文展开点 -> 资料边界。

迁移规则：

- 不要一上来先写大而空的行业背景。
- 不要把单一对象的调研写成宽泛综述。
- 不要在第 2、3、4 节连续重复“它本质上是一个……”“它更像一层……”这类同层级定义。前文已经立住的定位，后文默认继承即可。

### 2. 结论要早点给，但必须带边界

- 调研报告通常应该较早给出 findings snapshot，让读者先知道大致判断。
- 但结论必须同时带出前提、范围和适用条件。
- 好结论不是“它很好”或“它不行”，而是“在什么目标下它成立，在什么目标下它不成立”。
- 如果对象比较复杂，先给 recommendation tier，再解释机制，会比先给抽象定义更稳。

迁移规则：

- 不要把真正结论拖到结尾。
- 不要写绝对化判断，除非证据边界足够强。
- 不要让“架构上更像 X”代替“采用上该怎么判断”。

### 3. 证据、判断、建议要分层

- 源材料说了什么，是一层。
- 结合多份资料后的综合判断，是一层。
- 面向读者的建议、采用建议或 next step，是第三层。
- 这三层可以连着写，但不能混成一个模糊段落。

迁移规则：

- 避免“资料里写了 A，所以一定应该选 A”这种直跳。
- 更自然的推进方式是：从资料看什么，综合起来意味着什么，因此建议什么。
- 但不要把“我是如何分析这些资料的”写到正文表面。把分析过程转成对象陈述和判断陈述，而不是写成“更稳的做法是……”“这句话需要拆开看……”这类方法说明。

### 4. 必须给出一个最小 SOP

- 调研报告不能只停在“理解对象”，还要回答“怎么最快验证”。
- 这里的重点是 first-pass 路径，不是完整教程。
- 好的最小 SOP 应该把前置条件、关键动作和最先验证的目标压缩清楚。
- 这一节的标题也尽量像真实读者会问的问题，例如“怎么开始试一轮 X”“先怎么试起来”“第一步先做什么”，而不是“最小验证路径：先验证什么最有意义”或“如果要试一轮，最短路径是什么”这类仍然带作者分析痕迹的标题。
- 如果已经有真实尝试或 deployment 记录，不要把这些内容埋进 SOP 说明里。更自然的做法是单独给一小段 `实测`：试了什么、走到了哪一步、卡在什么地方、这对判断意味着什么。

迁移规则：

- 不要把 quickstart 展开成操作手册。
- 也不要只讲概念不讲验证入口。

### 5. 场景边界比能力清单更重要

- 调研报告不只是解释“它能做什么”，还要解释“在什么场景值得选，在什么场景不值得选”。
- 最有价值的判断通常来自场景边界，而不是能力列表。
- 如果不写不适用场景，很多“推荐”其实没有判断力。
- 结论摘要里最好直接把 `更适合 / 不适合` 压缩出来，而不是等到正文后半段才出现。

迁移规则：

- 把抽象优点改写成对象化判断。
- 如有必要，单独给一小段“不适合”的说明。

### 6. 约束、坑和成本要进入正文主线

- 真正影响决策的 often 不是能力上限，而是配置成本、运维复杂度、长链路稳定性、可观察性、调试难度和回退成本。
- 这些信息不能后置成脚注，也不该只在结尾顺手带过。
- 如果存在真实失败点、环境限制或踩坑记录，例如安装失败、动态库限制、模型/backend 不兼容、权限问题，这些信息通常比一串功能清单更有判断价值，应当在正文里被明确写出。

迁移规则：

- 如果资料里已经提到 limitations、assumptions、failure mode、cost 或 guardrails，把它们前置到评估章节。
- 把“有局限”改成“局限体现在哪个链路、哪个前提、哪个操作层面”。

### 7. 对复合卖点先拆解，再判断

- 很多项目会用一句复合定位句同时表达部署方式、硬件门槛、能力范围、兼容面和支持边界。
- 这类句子通常很关键，但不能直接整句复述成结论。
- 更好的方式是先拆成若干 adoption 维度，再分别判断。

常见维度包括：

- 部署门槛
- 能力广度
- 兼容性
- 扩展方式
- 运行前提与配置成本
- 支持边界

迁移规则：

- 不要把复合卖点原样翻译成一段中文判断。
- 要把“官方怎么说”“资料能支持到哪里”“哪些点仍未验证”拆开写。
- 但在 reader-facing 正文里，少写“这句口号应该怎么拆”这类方法论表达。更自然的写法是直接进入对象判断，例如直接说“接口兼容主要体现在哪一层”“多模型支持主要依赖什么”“哪些能力还带明显配置成本”。

### 8. 对开源项目，源码追踪是必要环节

- 对开源或 source-available 项目，读者往往不只关心“官方怎么说”，还关心“实现是否大致支撑这个说法”。
- 因此，源码追踪应视为调研流程的一部分，而不是可选增强。
- 这里要求的是 selective code reading，不是全量 repo audit。
- 目标是看清几个最影响采用判断的实现锚点，而不是把报告写成源码漫游。

迁移规则：

- 不要完全停在 docs 摘要。
- 也不要把整个正文写成 repo walkthrough。
- 对工具型、平台型项目，源码阅读更适合做“关键卖点背后的证据层”，而不是整篇正文的主叙事。

### 9. 对关键卖点，要补一层实现形态解释

- 对开源项目，很多判断不止于“是否支持”，还包括“靠什么实现”“是否需要显式配置”“是不是某种组合路径”。
- 当一个卖点明显影响采用判断时，正文最好补一层简短实现解释。
- 这种解释不需要深入每个源码细节，但最好能回答：
  - 它更像内建能力、统一控制面，还是组合出来的能力
  - 它是否依赖显式配置、插件注册、扩展点或特定 pipeline
  - 对使用者来说，它是开箱即用的 first pass，还是带明显配置成本的能力

迁移规则：

- 不要把“文档写着支持”直接翻成“具备稳定能力”，却不解释实现形态。
- 如果当前只做到 docs-level 判断，就明确写出来。

### 10. 观察和评估要分开

- `观察` 更接近事实层：看到哪些入口、能力面、配置方式、UI、文档承诺、代码结构或实测现象。
- `评估` 更接近判断层：这些事实说明它适合什么、不适合什么、成本主要落在哪。
- 这两层可以前后紧挨着写，但不要在同一句里来回切。

迁移规则：

- 不要把“我看到了什么”和“因此它很好/不好”压成一个混杂段。
- 如果材料比较杂，先列 3 到 5 个关键观察，再收成一个更短的评估段，读感通常会更顺。

### 11. 产品层优先，实现层支撑

- 对具名工具、平台或服务型项目，正文优先回答“你会怎么接触到它”“它提供了哪些使用面”“这些使用面对采用意味着什么”。
- 实现层内容仍然重要，但更适合作为对关键判断的支撑，而不是整篇正文的主组织方式。
- 如果正文很早就进入文件路径、路由名、配置字段，而读者还没先建立产品层理解，读感通常会变得很硬。

迁移规则：

- 优先写产品定位、入口、能力面、观察、场景与风险。
- 把实现层解释压缩到真正影响判断的点上，例如兼容性到底落在哪一层、多模态是内建还是组合、扩展是不是正式设计点。
- 不要让“源码证明”压过“项目到底好不好理解、好不好试、值不值得继续看”。

## 文风要求

### 1. 语气更像 memo，而不是文章

- 项目调研的读者通常不是来读叙事，而是来做判断。
- 因此语气应该直接、边界清晰、围绕判断问题拆段。
- 每段最好只解决一个判断问题，不要把定位、优点、建议混成一个大段。
- 对于结论摘要，优先写成“判断卡片”而不是“抽象总论”。
- 但“判断卡片”不等于流水账式编号。能收成短段落时，优先用短段落。
- 真正成熟的调研文，判断维度存在，但语言表面不应该充满模板感。要像自然说服，而不是条款罗列。
- 对具名工具或平台型项目，允许出现一句很克制的对象类比，例如“更像一层私有 API 服务”“更像统一入口”“像一个万能插座”。但类比只能帮助建立直觉，后面仍然要回到事实和判断本身。

### 2. 句式要沿着“问题 -> 证据 -> 判断”推进

- 句子可以偏中长，但推进关系要稳定。
- 优先使用能体现边界和证据层级的表达。

更自然的表达通常是：

- 本次调研聚焦……
- 从当前资料看……
- 如果目标是……
- 更适合……
- 不太适合……
- 值得先做一轮 PoC / 当前更适合观望……
- 综合这些信息……
- 如果目标是 first pass 验证……
- 若要进入更高要求场景……
- 这一步真正需要注意的是……

迁移规则：

- 同一小节里避免三次以上使用同一种句式骨架。
- 少用“对这次调研来说”“如果你的目标是”“这意味着”这类明显的模板桥句连续起手。
- 允许把几个判断压缩成一个更顺的综合段，而不是拆成多条形式正确但读感僵硬的句子。
- 也不要走到另一端，把多个判断硬塞进一句长句里。对于结论段，宁可拆成三个短判断，也不要写成“它更适合……；如果只是……；当前……；还需要……”这一类回摆很重的句子。

### 3. 少修辞，多判断

- 不要为了“读起来像文章”而堆修辞。
- 研究报告最重要的是判断密度，不是抒情感或叙事感。
- 方法论可以有，但应该从资料和判断里长出来，而不是先行宣布。

## 与其他风格的边界

- 它不是 `practice-sharing`：不以个人或团队实践演进作为主叙事。
- 它不是 `serious-engineering`：不只解释机制，还要帮助读者完成选型、判断和试跑。
- 它不是 `rfc-document`：不默认面向评审和实施对齐，也不要求 proposal 章节。
- 它和 `editorial-analytic` 有交叉，但更收束、更证据驱动，也更强调 SOP、场景边界、坑和建议。

## 不要做的事

- 不要把调研报告写成泛泛摘要。
- 不要把调研报告写成个人经验贴。
- 不要只列能力和优点，不写边界、坑和成本。
- 不要把结论摘要写成只有架构抽象词、但没有 recommendation tier 和场景边界的“定义句”。
- 不要把单一来源写成大范围共识。
- 不要把一个具名项目自动扩成整个技术类别，除非用户明确要求。
- 不要把 quick SOP 扩成完整教程，导致正文失去判断密度。
- 不要把开源项目调研停在文档层面，却仍然给出像完成评估一样的口吻。
- 不要把 selective code reading 误做成 full repo audit。
- 不要让内部写作框架直接暴露在表面，例如把“对象定位 / recommendation tier / fit / first-pass / evidence boundary”逐项写成像填表一样的显式段落。

## 快速判断

如果改写后的文本满足下面这些特征，通常就更接近 research-report：

- 一开始就说明研究对象、问题和资料边界
- 较早给出有边界的关键结论，并且结论包含 recommendation tier
- 证据、判断、建议三层比较清楚
- 至少回答了：怎么快速验证、适合什么场景、主要坑在哪里
- 对开源项目，至少有一层源码视角支撑关键卖点判断
- 结尾能给出相对克制、可执行的建议或下一步
